# Spring Cloud Alibaba 分布式技术基座架构设计

## 1. 项目定位

本项目是一个基于 Spring Cloud Alibaba 的分布式技术验证实验室。

项目的建设顺序是：先建设可通过 Maven 复用的技术基座；再组合基座建设通用分布式能力；最后由业务场景验证复杂链路中的治理、一致性、可靠性和恢复语义。

```text
技术基座层
  -> 领域能力层
    -> 综合业务场景层
```

| 层级 | 目标 | 业务语义 |
|---|---|---|
| 技术基座层 | 技术封装、性能、协议、Adapter、Starter、快速接入、可替换实现 | 不包含 |
| 领域能力层 | 幂等、可靠消息、缓存一致性、事务、Saga、韧性、调度等通用分布式问题 | 仅限通用技术语义 |
| 综合业务场景层 | 订单、支付、库存、履约等业务链路的串联 | 包含 |

## 2. 核心原则

### 2.1 技术基座优先

缓存、消息、协调、服务治理、数据、安全、可观测性都是独立技术领域。它们不能依赖订单、支付、库存等业务模型，也不能只为某个业务场景存在。

### 2.2 Maven 积木化

每个技术基座必须同时提供可复用库产物和独立实验应用：

```text
技术基座 = contract + core + adapter + starter + lab
```

| 产物 | 职责 |
|---|---|
| `contract` | 稳定接口、公共 DTO、SPI、配置模型、技术协议 |
| `core` | 不依赖具体中间件的策略、状态机、默认实现 |
| `adapter` | Redis、RocketMQ、Nacos、Seata 等具体技术实现 |
| `starter` | Spring Boot 自动装配、默认配置、指标、异常收敛和上下文传播 |
| `lab` | 独立验证技术本身的应用，上层禁止依赖 |

上层场景通过 Maven 引用 `contract` 和 `starter`，不引用 `lab`。

### 2.3 上层不直接依赖中间件 SDK

上层能力和场景默认不直接依赖 `RocketMQTemplate`、`RedissonClient`、`NacosConfigManager` 等具体 SDK，而是使用基座提供的稳定接口。

需要专属技术语义时，依赖显式能力接口，例如：

```text
OrderedEventPublisher
DelayedEventPublisher
TransactionalEventPublisher
FencingLock
LeaderElector
```

### 2.4 兼容不掩盖语义差异

公共协议只统一稳定部分。RocketMQ 事务消息、Kafka Offset、RabbitMQ Confirm 的语义不等价，必须通过显式能力接口与原生 Adapter 暴露，不能伪造万能接口。

### 2.5 Spring Cloud Alibaba 主线

- Nacos：配置与注册发现。
- Sentinel：流量控制、熔断、系统保护。
- Seata：AT/TCC 强一致事务。
- RocketMQ：首个、最完整的消息实现。
- Spring Cloud Stream RocketMQ Binder：便携消息发布订阅实现。

Kafka、RabbitMQ、Redisson、ZooKeeper、Resilience4j、Spring Authorization Server、ShardingSphere、OpenTelemetry 等作为特定领域的对比或扩展实现。

## 3. Maven 总体结构

```text
spring-cloud-alibaba-distributed-lab
├── pom.xml
│
├── platform-foundation
├── platform-governance
├── platform-message
├── platform-cache
├── platform-coordination
├── platform-data
├── platform-security
├── platform-observability
├── platform-fault
│
├── capability-command
├── capability-reliable-message
├── capability-cache-consistency
├── capability-transaction
├── capability-saga
├── capability-resilience
├── capability-scheduler
├── capability-data-projection
│
└── scenarios
    ├── scenario-order
    ├── scenario-payment
    ├── scenario-inventory
    └── scenario-fulfillment
```

依赖方向严格单向：

```text
platform-* -> capability-* -> scenarios/*
```

禁止：

1. `platform-*` 依赖 `capability-*` 或 `scenarios/*`。
2. `capability-*` 依赖具体业务场景。
3. 场景模块复制消息、缓存、锁、追踪、幂等实现。
4. `contract` 引入具体 Broker、缓存、注册中心或业务模块依赖。
5. 同层模块产生实现级双向依赖。

## 4. 第一层：技术基座

技术基座只面向技术本身，目标是统一封装、优化性能、提供稳定接口、提供 Starter、降低接入成本，并隐藏连接、序列化、上下文传播、指标与异常处理细节。

### 4.1 platform-foundation

```text
platform-foundation
├── foundation-contract
├── foundation-context
├── foundation-idempotency
├── foundation-id
└── foundation-starter
```

提供：

- `RequestContext`、`TraceContext`、`PrincipalContext`。
- `IdempotencyKey`、`CommandId`、`EventId`、`AggregateId`。
- 统一错误码与异常模型。
- 时间、版本、乐观并发控制约定。
- MVC、WebFlux、异步执行器中的上下文传播。

不包含 Redis、Nacos、RocketMQ、Seata SDK，也不包含业务实体、Mapper、Repository 和 Controller。

### 4.2 platform-governance

```text
platform-governance
├── governance-contract
├── nacos-config-starter
├── nacos-discovery-starter
├── gateway-starter
├── sentinel-starter
├── rpc-openfeign-starter
└── governance-lab
```

提供：

- Nacos 配置加载、命名空间、Group、DataId 命名规范。
- 服务注册、发现、实例元数据、健康检查、优雅上下线。
- 元数据模型：`version`、`zone`、`region`、`capability`。
- Gateway 路由、灰度、限流、签名、防重放扩展点。
- OpenFeign 超时、Trace、身份、灰度标签传播。
- Sentinel 规则、Block Handler、动态规则来源约定。

### 4.3 platform-message

消息基座支持多 Broker。RocketMQ 是 Spring Cloud Alibaba 主线的首个完整实现，但不是唯一目标。

```text
platform-message
├── message-contract
├── message-core
├── message-stream-starter
├── message-rocketmq-adapter
├── message-rocketmq-starter
├── message-kafka-adapter
├── message-kafka-starter
├── message-rabbitmq-adapter
├── message-rabbitmq-starter
└── message-lab
```

#### message-contract

定义不依赖 Broker 的事件信封：

```text
eventId
eventType
schemaVersion
producer
aggregateType
aggregateId
partitionKey
idempotencyKey
occurredAt
traceparent
headers
payload
```

```java
public record EventEnvelope<T>(
        String eventId,
        String eventType,
        int schemaVersion,
        String producer,
        String aggregateType,
        String aggregateId,
        String partitionKey,
        String idempotencyKey,
        Instant occurredAt,
        Map<String, String> headers,
        T payload
) {
}
```

#### message-core

提供事件序列化、Schema Registry、Upcaster、Topic/Group 命名治理、重试、死信、回放策略，以及 Trace/日志/指标字段统一。

Outbox 和 Inbox 属于可靠消息能力，放在第二层 `capability-reliable-message`。

#### Broker Adapter 分工

| Adapter | 专属验证能力 |
|---|---|
| RocketMQ | 顺序、延迟、事务消息、Tag/Key、重试、DLQ、消息查询 |
| Kafka | Partition、Offset、Rebalance、幂等/事务生产者、Compaction、Replay |
| RabbitMQ | Exchange、Routing Key、Confirm/Return、ACK/NACK、TTL、DLX、Prefetch |
| Spring Cloud Stream | 公共发布订阅、Binding、Destination、Group、Binder 可替换性 |

显式能力接口：

```text
PortableEventPublisher
OrderedEventPublisher
DelayedEventPublisher
TransactionalEventPublisher
ReplayableEventConsumer
```

### 4.4 platform-cache

```text
platform-cache
├── cache-contract
├── cache-core
├── cache-redis-starter
├── cache-caffeine-starter
├── cache-multilevel-starter
└── cache-lab
```

提供 Cache Key、Value、Version、TTL、失效事件协议，以及 Redis、Caffeine、二级缓存、序列化、空值、TTL、异常策略、命中率、热 Key 和重建耗时指标。

不直接实现商品、库存、用户等业务缓存。

### 4.5 platform-coordination

```text
platform-coordination
├── coordination-contract
├── coordination-redisson-starter
├── coordination-zookeeper-starter

```

提供：

```text
DistributedLock
Lease
FencingToken
LeaderElector
WorkClaim
```

Redisson 是主实现；ZooKeeper/Curator 用于对比临时节点、Watcher、Session 和选主语义；XXL-Job 只提供调度、广播分片和重试执行适配，不替代锁、租约与 Fencing Token。

### 4.6 platform-data

```text
platform-data

├── data-flyway-starter
├── data-shardingsphere-starter
├── data-mongodb-starter

```

提供 MyBatis-Flex、Mapper XML、Flyway、ShardingSphere-JDBC、MongoDB、分布式 ID、数据源与事务边界约定。基座不出现用户、订单、库存等表模型。

### 4.7 platform-security

```text
platform-security
├── security-contract
├── security-sa-token-starter
├── security-oidc-starter
├── security-resource-server-starter
└── security-lab
```

提供 `PrincipalContext`、Token 提取/传播、授权决策接口、Sa-Token + Redis、OAuth2/OIDC/JWT/JWKS、Resource Server、服务间身份与安全审计。

Sa-Token 与 OAuth2/OIDC 作为身份 Adapter 并存；上层仅依赖统一身份协议。

### 4.8 platform-observability

```text
platform-observability
├── observability-contract
├── observability-otel-starter
├── observability-logging-starter
├── observability-metrics-starter
└── observability-lab
```

固定采用：

```text

```

### 4.9 platform-fault

```text
platform-fault
├── fault-contract
├── fault-switch-starter

```

提供应用内延迟、异常、重复执行、中断点，及 Toxiproxy 网络延迟、断连、带宽限制能力；不定义具体业务故障。

## 5. 第二层：领域能力

领域能力层组合技术基座，解决通用分布式问题。允许出现命令状态、缓存一致性状态、Outbox 事件状态、Saga 状态机等通用技术语义，但不得出现订单、支付、库存等具体业务模型。

```text
能力模块 = contract + core + starter + lab
```

### 5.1 capability-command

```text
capability-command
├── command-contract
├── command-idempotency-core
├── command-idempotency-redis-starter
└── command-lab
```

组合 `platform-foundation`、`platform-cache`、`platform-observability`，解决命令去重、幂等键占位、执行中/成功/失败状态机、并发抢占、响应缓存和重放。

### 5.2 capability-reliable-message

```text
capability-reliable-message
├── reliable-message-contract
├── outbox-core
├── inbox-core
├── outbox-mybatis-starter
├── inbox-mybatis-starter
├── outbox-dispatcher-starter
└── reliable-message-lab
```

组合 `platform-data`、`platform-message`、`platform-coordination`、`platform-observability`，解决 Transactional Outbox、Inbox 去重、至少一次投递下的幂等副作用、事件抢占/租约/重试、DLQ、回放、对账与恢复。

### 5.3 capability-cache-consistency

```text
capability-cache-consistency
├── cache-consistency-contract
├── cache-aside-starter
├── cache-invalidation-starter
├── hot-key-starter
├── logical-expiration-starter
└── cache-consistency-lab
```

组合 `platform-cache`、`platform-coordination`、`platform-message`、`platform-observability`，解决 Cache Aside、双删/延迟双删、失效事件、缓存穿透、击穿、雪崩、热 Key、逻辑过期、一致性修复和 Redis 故障降级。

### 5.4 capability-transaction

```text
capability-transaction
├── transaction-contract
├── seata-at-starter
├── seata-tcc-starter
└── transaction-lab
```

组合 `platform-governance`、`platform-data`、`platform-observability`、`platform-fault`，验证 Seata AT、TCC、Confirm/Cancel 幂等、空回滚、防悬挂和事务恢复。

### 5.5 capability-saga

```text
capability-saga
├── saga-contract
├── saga-state-machine-core
├── saga-mybatis-starter
├── saga-message-starter
├── saga-recovery-starter
└── saga-lab
```

组合 `platform-data`、`platform-message`、`platform-coordination`、`platform-observability`，解决长事务最终一致性、正向动作和补偿、补偿幂等、补偿失败重试、人工恢复和对账。

### 5.6 capability-resilience

```text
capability-resilience
├── resilience-contract
├── timeout-budget-starter
├── retry-starter
├── bulkhead-starter
├── circuit-breaker-starter
└── resilience-lab
```

组合 `platform-governance`、`platform-observability`、`platform-fault`，解决 Deadline、超时预算、限次重试、指数退避、抖动、重试风暴防护、Bulkhead、熔断、降级和过载保护。

Sentinel 与 Resilience4j 可以作为对比实现，能力层不暴露具体框架 API。

### 5.7 capability-scheduler

```text
capability-scheduler
├── scheduler-contract
├── scheduler-xxl-job-starter
├── work-claim-starter
├── reconciliation-starter
└── scheduler-lab
```

组合 `platform-coordination`、`platform-data`、`platform-observability`，解决分片任务、单实例任务、任务抢占、失败重试、对账、补偿和任务幂等。

### 5.8 capability-data-projection

```text
capability-data-projection
├── projection-contract
├── projection-consumer-starter
├── projection-mysql-starter
├── projection-mongodb-starter
├── projection-reconciliation-starter
└── projection-lab
```

组合 `platform-data`、`platform-message`、`platform-observability`，解决事件驱动读模型、MySQL/MongoDB 投影、投影幂等、延迟、漂移、对账、重建和修复。

## 6. 第三层：综合业务场景

```text
scenarios
├── scenario-order
├── scenario-payment
├── scenario-inventory
└── scenario-fulfillment
```

场景层只引用技术基座与领域能力 Starter。例如：

```text
scenario-order
├── foundation-starter
├── nacos-discovery-starter
├── gateway-starter
├── security-sa-token-starter
├── rpc-openfeign-starter
├── command-idempotency-redis-starter
├── seata-at-starter
├── outbox-mybatis-starter
├── message-rocketmq-starter
├── cache-invalidation-starter
├── observability-otel-starter
└── fault-switch-starter
```

场景层只描述业务流程和高级状态不变量，例如：

- 重复请求时，订单、库存和付款不能重复执行。
- 本地事务提交后，业务事件最终必须可靠投递。
- 缓存失效失败时，读取路径不能长期返回错误状态。
- 下游超时后，业务状态必须可查询、可恢复、可补偿。
- 异步消费重复时，读模型不能出现重复副作用。

场景层不得自行实现幂等状态机、Outbox/Inbox、消息 Header、缓存失效、分布式锁或 Trace 透传。

## 8. 技术选型基线

| 领域 | 主技术选型 | 对比或扩展 |
|---|---|---|
| Java | Java 21 LTS | Java 17 兼容性按需验证 |
| Spring | Spring Boot 3.5.x、Spring Cloud 2025.0.x | 当前采用稳妥基线；Boot 4 / Spring Cloud 2025.1.x 后续逐步升级 |
| Spring Cloud Alibaba | 2025.0.x | Nacos、Sentinel、Seata、RocketMQ 主线 |
| 消息 | RocketMQ + Spring Cloud Stream | Kafka、RabbitMQ |
| 缓存 | Redis + Caffeine | 多级缓存 |
| 协调 | Redisson | ZooKeeper/Curator |
| 调用韧性 | Sentinel | Resilience4j |
| 数据 | MySQL、MyBatis-Flex、Flyway、ShardingSphere、MongoDB | CDC/Debezium 后置 |
| 调度 | XXL-Job | - |
| 身份 | Sa-Token + Redis | Spring Authorization Server + OAuth2/OIDC/JWT |
| 可观测性 | OpenTelemetry + LGTM | - |
| 故障注入 | Fault SPI + Toxiproxy | - |

## 9. 实施顺序

1. 创建根 POM、三层聚合结构和统一包命名。
2. 建设 `platform-foundation`，确定上下文、标识、事件、幂等和错误协议。
3. 建设 `platform-observability` 和 `platform-fault` 的最小横切能力。
4. 建设技术基座：治理、消息、缓存、协调、数据、安全。
5. 建设领域能力：命令幂等、可靠消息、缓存一致性、事务、Saga、韧性、调度、数据投影。
6. 最后建设 `scenarios`，只组合成熟 Starter。
7. 技术能力稳定后，再补充部署配置、实验脚本、README、架构图和技术文档。

## 10. 完成标准

### 技术基座

1. 提供稳定 Contract。
2. 至少提供一个 Adapter 与 Starter。
3. Starter 自动完成装配、上下文传播、指标和异常收敛。
4. 存在独立 `lab` 验证技术行为。
5. 不依赖具体业务场景。
6. 关键性能、故障和恢复路径可独立验证。

### 领域能力

1. 只通过技术基座 Contract 和 Starter 组合能力。
2. 有明确状态机和状态不变量。
3. 包含并发、重复、延迟、超时或故障验证。
4. 对上层提供可复用 Starter。
5. 不出现订单、支付、库存等具体业务实体。

### 业务场景

1. 只依赖已完成的技术基座和领域能力 Starter。
2. 不复制底层技术实现。
3. 描述业务流程、高级分布式语义和业务状态不变量。
4. 能在故障注入下观察一致性、幂等、补偿和恢复结果。
