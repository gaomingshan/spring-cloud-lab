# Spring Cloud Alibaba 分布式技术基座技术选型

## 1. 选型目标

本技术选型服务于三层架构：

```text
技术基座层 -> 领域能力层 -> 综合业务场景层
```

选型原则：

1. Spring Cloud Alibaba 是服务治理与分布式中间件主线。
2. 每个技术领域先固定一个默认主实现，避免首期维护多套等价技术。
3. 需要比较不同分布式语义时，预留 Adapter，但不在首期全部实现。
4. 技术基座提供 Contract、Adapter、Starter 和独立 Lab；场景层不直接依赖中间件 SDK。
5. 技术版本由根 POM 和 BOM 统一管理，子模块不得自行覆写。

## 2. 固定运行时基线

| 分类 | 固定选择 | 版本 | 说明 |
|---|---|---:|---|
| JDK | Java LTS | 21 | 新项目统一 Java 21；虚拟线程可用于 RPC、消息消费与调度性能实验 |
| 构建工具 | Maven | 3.9+ | 多模块、BOM、Enforcer、依赖收敛 |
| Spring Boot | Spring Boot | 3.5.9 | 当前采用成熟稳妥的 Boot 3.5 基线；Boot 4 作为后续升级目标 |
| Spring Cloud | Spring Cloud | 2025.0.0 | 与 Spring Boot 3.5.x 对齐 |
| Spring Cloud Alibaba | Spring Cloud Alibaba | 2025.0.0.0 | 与 Spring Cloud 2025.0.x、Spring Boot 3.5.x、JDK 17+ 对齐 |
| ORM | MyBatis-Flex + MyBatis XML | 由根 POM锁定 | Flex 用于基础 CRUD；XML 用于条件更新、锁、批处理与复杂查询 |
| 数据库迁移 | Flyway + flyway-mysql | Spring Boot BOM 管理 | 所有持久化模块用版本化迁移，不维护手工初始化 SQL |
| 对象映射 | MapStruct | 根 POM 锁定 | DTO、Entity、Command、Event 显式转换 |
| 参数校验 | Jakarta Validation | Spring Boot BOM 管理 | HTTP、RPC、消息入口的边界校验 |

当前采用 Spring Cloud Alibaba 官方兼容矩阵中的稳妥组合：`2025.0.x` 对应 Spring Boot `3.5.x` 和 Spring Cloud `2025.0.x`。Spring Boot 4 / Spring Cloud 2025.1.x 暂不作为首期基线，待各个技术模块独立验证后再逐步升级。

## 3. 技术基座层

### 3.1 platform-foundation

| 能力 | 固定选择 |
|---|---|
| 请求上下文 | Spring `ThreadLocal` + Micrometer Context Propagation |
| Trace 标准 | W3C Trace Context |
| 异步上下文传播 | Micrometer Context Propagation |
| 幂等标识 | `Idempotency-Key`、`CommandId`、`EventId` |
| 分布式 ID | ShardingSphere Snowflake |
| 错误协议 | RFC 9457 Problem Details 风格 + 内部领域错误码 |

不将 TransmittableThreadLocal 设为默认基础约定；它可作为上下文传播对比实验。

### 3.2 platform-governance

| 技术域 | 固定选择 | 职责 |
|---|---|---|
| 配置中心 | Nacos Config | 配置、命名空间、Group、DataId、动态刷新 |
| 注册发现 | Nacos Discovery | 服务发现、实例元数据、健康、上下线 |
| API 网关 | Spring Cloud Gateway Reactive | 路由、版本、灰度、入口治理 |
| 客户端负载均衡 | Spring Cloud LoadBalancer | 实例选择 |
| 声明式 RPC | OpenFeign | 服务间 HTTP 调用 |
| HTTP 客户端 | Apache HttpClient 5 | 连接池、超时与连接管理 |
| 服务保护 | Sentinel | QPS、热点、系统保护、熔断规则 |
| Sentinel 规则中心 | Nacos DataSource | 规则版本化与动态下发 |

`Spring HTTP Interface` 仅作为 RPC 对比实现，不作为首期标准。

### 3.3 platform-message

消息基座采用“公共协议 + 多 Broker Adapter + 显式高级能力”的模型。

| 层次 | 固定选择 |
|---|---|
| 事件协议 | 自定义 `EventEnvelope<T>` |
| 首期事件编码 | JSON，payload 版本由客户端自行约定 |
| 公共消息抽象 | Spring Cloud Stream |
| 首期 Broker | RocketMQ |
| RocketMQ 专属能力 | 原生 RocketMQ Spring/Java Client Adapter |
| 第二阶段对比 | Kafka Adapter |
| 第三阶段对比 | RabbitMQ Adapter |
| Protobuf | 第二阶段，引入用于跨语言编码对比 |

消息能力接口固定为：

```text
PortableEventPublisher
OrderedEventPublisher
DelayedEventPublisher
TransactionalEventPublisher
ReplayableEventConsumer
```

| 技术 | 固定用途 |
|---|---|
| Spring Cloud Stream | 可移植发布订阅、Binder 验证 |
| RocketMQ | 默认主实现；顺序、延迟、事务、Tag/Key、重试、DLQ |
| Kafka | Partition、Offset、Rebalance、Compaction、Replay 对比 |
| RabbitMQ | Exchange、Routing Key、Confirm、ACK/NACK、TTL、DLX 对比 |

可靠生产、Outbox、Inbox 和消费幂等不属于消息基座，属于 `capability-reliable-message`。

### 3.4 platform-cache

| 能力 | 固定选择 |
|---|---|
| 分布式缓存 | Redis 8.x |
| Spring Redis 客户端 | Spring Data Redis + Lettuce |
| 本地缓存 | Caffeine |
| 二级缓存 | Caffeine + Redis |
| 缓存失效广播 | platform-message 事件总线 |
| 热 Key / 重建协调 | Redisson |
| 布隆过滤器 | 首期轻量 Contract；RedisBloom 后置 |

首期不引入 JetCache、Cache2k 等额外封装，直接验证 Redis、Caffeine 和二级缓存真实语义。

### 3.5 platform-coordination

| 能力 | 默认主实现 | 对比实现 |
|---|---|---|
| 分布式锁 | Redisson 4.0.0 | ZooKeeper/Curator |
| 租约与续租 | Redisson Watchdog | ZooKeeper Session |
| Fencing Token | Redis `INCR` | ZooKeeper 临时顺序节点 |
| Leader Election | Redisson | Curator LeaderSelector |
| 调度中心 | XXL-Job | - |
| 分片任务 | XXL-Job 广播分片 | 自定义 `WorkClaim` |
| 限流 | Redisson RateLimiter | Sentinel 对比 |

Redisson 是默认协调技术，原因是它与 Redis 缓存、网关限流、Session 状态天然共用。ZooKeeper 仅用于协调语义对比，不作为首期全局基础依赖。

### 3.6 platform-data

| 能力 | 固定选择 |
|---|---|
| 主关系数据库 | MySQL 8.4 LTS |
| 数据访问 | MyBatis-Flex + MyBatis XML |
| Schema 管理 | Flyway + flyway-mysql |
| 分库分表 | ShardingSphere-JDBC |
| 分布式 ID | ShardingSphere Snowflake |
| 文档数据库 | MongoDB 8.x |
| 数据投影 | MySQL Projection / MongoDB Projection |
| 数据对账 | XXL-Job + MyBatis-Flex |
| CDC | 后置：Debezium + Kafka |

ShardingSphere 首期用于分片、读写分离、路由和分布式 ID。分布式强一致事务由 Seata 能力模块负责，不将 ShardingSphere XA 作为首期事务主线。

### 3.7 platform-security

身份体系采用双 Adapter，不做二选一。

| 子域 | 固定选择 | 用途 |
|---|---|---|
| 轻量共享会话 | Sa-Token + Redis | 多端登录、Session、Token 撤销、网关认证 |
| 标准身份协议 | Spring Authorization Server | OAuth2.1、OIDC、JWT、JWKS、标准 Token 生命周期 |
| 资源服务 | Spring Security Resource Server | JWT 验签、Audience、Scope、资源鉴权 |
| 服务身份 | OAuth2 Client Credentials | 服务间身份与权限 |
| 授权模型 | Spring Security Method Security | RBAC、权限码、资源授权 |
| 密码与密钥 | BCrypt + JWK | 密码存储、JWT 签名和密钥轮换 |

Sa-Token 负责 Spring Cloud Alibaba 风格共享登录态实验；OAuth2/OIDC 负责标准协议和服务间身份实验。上层仅依赖 `PrincipalContext` 和授权接口。

### 3.8 platform-observability

| 能力 | 固定选择 |
|---|---|
| Trace | OpenTelemetry Java Agent 为主；SDK 用于自定义 Span |
| Metrics | Micrometer + OTLP Export |
| Logs | Logback JSON + MDC |
| Collector | Grafana Alloy |
| Trace Store | Tempo |
| Metrics Store | Mimir |
| Log Store | Loki |
| UI | Grafana |

唯一默认可观测性路线：

```text
OpenTelemetry
  -> Grafana Alloy
    -> Tempo / Mimir / Loki
      -> Grafana
```

统一关联字段：

```text
trace_id
span_id
request_id
event_id
aggregate_id
idempotency_key
consumer_group
message_destination
fencing_token
```

不维护 SkyWalking、ELK、单独 Prometheus/Grafana 栈作为并行主线。

### 3.9 platform-fault

| 能力 | 固定选择 |
|---|---|
| 应用内故障 | 自建 Fault SPI + Spring Boot Starter |
| 网络故障 | Toxiproxy |
| Java Client | `eu.rekawek.toxiproxy:toxiproxy-java:2.1.7` |
| 故障类型 | 延迟、抖动、断连、带宽限制、重复执行、异常、中断点 |
| 故障标记 | OTel Span Attribute + Metric Tag + Log MDC |

## 4. 领域能力层

### 4.1 capability-command

| 能力 | 固定选择 |
|---|---|
| 命令幂等 | Redis 状态机 |
| 原子抢占 | Redis Lua Script |
| 命令状态 | `PROCESSING`、`SUCCEEDED`、`FAILED` |
| 响应重放 | Redis 保存标准响应摘要或业务结果引用 |
| 过期控制 | TTL + 业务状态校验 |

### 4.2 capability-reliable-message

| 能力 | 固定选择 |
|---|---|
| 生产可靠性 | Transactional Outbox |
| Outbox 存储 | MySQL + MyBatis XML |
| 分发抢占 | 数据库租约 Compare-And-Set |
| Broker 接入 | platform-message Contract |
| 消费去重 | Transactional Inbox / Processed Event Store |
| 重试 | 指数退避 + 最大次数 |
| 终态失败 | DLQ |
| 回放 | 受控 Replay Job |
| 对账 | XXL-Job |

固定语义：

```text
At-least-once delivery
  + Idempotent side effect
  = Exactly-once effect
```

不追求端到端 Exactly Once。

### 4.3 capability-cache-consistency

| 能力 | 固定选择 |
|---|---|
| 主读取模式 | Cache Aside |
| 更新后处理 | 删除缓存，不直接更新缓存 |
| 一致性增强 | 延迟双删 + 缓存失效事件 |
| 热点重建 | Redisson 锁 / SingleFlight |
| 逻辑过期 | Redis + Caffeine 二级缓存策略 |
| 穿透防护 | BloomFilter + 空值缓存 |
| 雪崩防护 | TTL 随机抖动 + 限流降级 |
| 修复 | XXL-Job 对账 / 重建任务 |

### 4.4 capability-transaction

| 能力 | 固定选择 |
|---|---|
| AT | Seata AT |
| TCC | Seata TCC |
| AT 适用范围 | 短事务、关系型数据库、Seata 代理数据源 |
| TCC 适用范围 | 资源预留、跨中间件、需要明确控制的事务 |
| 事务协调 | Seata Server |
| 注册与配置 | Nacos |
| 失败验证 | platform-fault |

### 4.5 capability-saga

| 能力 | 固定选择 |
|---|---|
| Saga 类型 | 编排式 Saga |
| 状态机 | 持久化 Saga State Machine |
| 存储 | MySQL |
| 步骤通信 | platform-message |
| 补偿 | 明确 Compensation Handler |
| 恢复 | XXL-Job |
| 对账 | 事件表与状态表比对 |

首期不引入 Camunda、Temporal 等工作流产品，避免将 Saga 语义验证替换成工作流产品学习。

### 4.6 capability-resilience

| 能力 | 固定选择 | 职责 |
|---|---|---|
| 服务端流控与熔断 | Sentinel | QPS、热点、系统保护、动态规则 |
| 客户端超时/重试 | Resilience4j | Timeout、Retry、退避、抖动 |
| 隔离 | Resilience4j Bulkhead | 信号量/线程池隔离 |
| 服务调用 | OpenFeign + Apache HttpClient 5 | Deadline 传播与连接池 |
| 故障观察 | OpenTelemetry + Toxiproxy | 重试、熔断和恢复观察 |

### 4.7 capability-scheduler

| 能力 | 固定选择 |
|---|---|
| 调度中心 | XXL-Job |
| 任务幂等 | capability-command |
| 执行权 | platform-coordination |
| 分片 | XXL-Job 广播分片 |
| 补偿/对账 | XXL-Job + MyBatis-Flex |
| 任务恢复 | 重试 + 对账扫描 |

### 4.8 capability-data-projection

| 能力 | 固定选择 |
|---|---|
| 消息来源 | platform-message |
| 投影幂等 | Inbox / Processed Event Store |
| 关系读模型 | MySQL |
| 文档读模型 | MongoDB |
| 重建 | 消息回放 + 全量校验 |
| 漂移修复 | XXL-Job 对账任务 |

## 5. 首期固定范围

首期只建设下列默认实现：

```text
Nacos
Sentinel
Spring Cloud Gateway
OpenFeign + LoadBalancer + Apache HttpClient 5
RocketMQ + Spring Cloud Stream RocketMQ Binder
Redis + Lettuce + Redisson
Caffeine
Seata AT / TCC
MySQL + MyBatis-Flex + Flyway
ShardingSphere-JDBC
MongoDB
XXL-Job
Sa-Token + Redis
OpenTelemetry + LGTM
Toxiproxy
```

以下技术只预留 Adapter 位置，放入后续对比阶段：

```text
Kafka
RabbitMQ
ZooKeeper
Spring Authorization Server
Debezium
Spring HTTP Interface
Protobuf
RedisBloom
```

这样可以先把 Spring Cloud Alibaba 主线做深，而不是同时维护多套中间件。

## 6. 根 POM 版本治理规则

1. Spring Boot、Spring Cloud、Spring Cloud Alibaba 只能通过 BOM 管理。
2. Boot BOM 管理的组件不手动指定版本。
3. Spring Cloud Alibaba 管理的 Nacos、Sentinel、Seata、Stream RocketMQ 依赖不手动指定版本。
4. 非 BOM 组件统一由根 `properties` 锁定。
5. 每个具体 Broker Client 只存在于对应 `platform-message/*-adapter`。
6. 场景模块只能依赖平台和能力 Starter，不能新增底层中间件 Client。
7. Maven Enforcer 负责 Java 21、Maven 3.9+、依赖收敛和重复类检查。
8. Flyway 的版本由 Spring Boot BOM 管理，同时显式引入 `flyway-mysql`。
9. Redisson 固定为 `4.0.0`。
10. Toxiproxy Java Client 固定为 `2.1.7`。

## 7. 首批 Maven 模块

技术选型固定后，首批只建立以下技术基座，不立即创建业务场景：

```text
platform-foundation
platform-observability
platform-fault
platform-governance
platform-message
platform-cache
platform-coordination
platform-data
```

首批基座均具备 Contract、Adapter、Starter 与独立 Lab 后，才进入领域能力层建设。
