# 分布式技术栈推进方案

## 运行模型

应用服务默认在本机通过 Maven 启动；中间件运行在 Docker 或局域网环境。两种方式都有效，关键是配置不再写死在代码库中。

所有新建或正在完善的模块使用以下环境变量，未设置时连接本机 Docker：

| 变量 | 默认值 | 说明 |
|---|---|---|
| `LAB_NACOS_ADDR` | `127.0.0.1:8848` | Nacos 地址 |
| `LAB_NACOS_NAMESPACE` | 空 | Nacos 命名空间 |
| `LAB_MYSQL_HOST` | `127.0.0.1` | MySQL 主机 |
| `LAB_MYSQL_PORT` | `3306` | MySQL 端口 |
| `LAB_MYSQL_USERNAME` | `root` | MySQL 用户 |
| `LAB_MYSQL_PASSWORD` | `root123` | MySQL 密码 |
| `LAB_ROCKETMQ_NAMESRV` | `127.0.0.1:9876` | RocketMQ NameServer |
| `LAB_ENV` | `local` | 指标环境标签 |

Seata TC 不放入通用 `docker-compose.yml`。TC 需要单独维护自己的持久化元数据、注册和事务组映射，避免用缺少元数据表或错误注册地址的临时容器制造不可复现的事务环境。

局域网中间件只需在启动终端设置上述变量。应用不需要容器化，Docker Compose 也不需要绑定某台特定宿主机 IP。

## 业务主线

业务主线是下单：网关接收请求，`commerce-order` 做幂等控制和 Seata 全局事务，调用库存与账户服务，随后通过 Outbox 投递订单事件。下游事件消费者、补偿任务、分库分表查询和异构数据投影在后续模块逐步接入。

```text
Client -> Gateway -> commerce-order -> storage/account
                    |                 |
                    |                 +-- Seata AT
                    +-- order + outbox -> RocketMQ -> projections/notifications
```

## 模块职责

| 模块 | 独立技术核心 | 业务主线中的验证点 |
|---|---|---|
| `lab-config-center` | Nacos 配置刷新与隔离 | 路由、限流、开关与业务参数的动态变更 |
| `lab-registry` | 注册、发现、实例元数据 | 实例上下线、权重与灰度流量 |
| `lab-gateway` | 鉴权、限流、路由、幂等 | 下单入口的防重放、用户级限流与灰度 |
| `lab-feign` | 超时、负载均衡、上下文透传 | 订单到下游服务调用 |
| `lab-sentinel` | 流控、熔断、规则持久化 | 保护下单、库存与账户依赖 |
| `lab-seata` | AT/TCC 事务语义 | 库存、账户与订单的一致性 |
| `lab-commerce` | 业务编排、幂等、Outbox | 贯通全部已成熟模块 |
| `lab-stream` | 消息投递、消费幂等、DLQ | OrderCreated 事件的可靠消费 |
| `lab-xxljob` | 重试、补偿、分片 | Outbox 补偿与不一致数据对账 |
| `lab-sharding` | 分片路由与查询限制 | 订单历史读模型 |
| `lab-multi-datasource` | 异构数据投影与最终一致 | 用户行为与订单检索投影 |
| `lab-security` | 身份、权限、服务间信任 | 网关身份传递和后台运维权限 |
| `lab-observability` | OTel + LGTM | Trace、日志与指标定位故障 |

## 推进顺序

1. 完成并稳定本地服务与 Docker 中间件的配置契约。
2. 以 `lab-commerce` 贯通 Nacos、Feign、Seata、RocketMQ Outbox。
3. 给 Gateway 接入下单路由、身份、限流和防重放；给 Sentinel 增加版本化规则。
4. 将 `OrderCreated` 消费、幂等、重试和死信完整落在 `lab-stream`。
5. 以 XXL-Job 实现 Outbox 补偿和订单投影对账。
6. 以 ShardingSphere 与多数据源实现读模型和数据分治。
7. 在每个阶段使用 OTel + LGTM 保留成功、失败和恢复的观测证据。

## 实验准则

每个模块只讨论一个技术核心，但必须提供：正常路径、一个可控失败路径、可观察的状态变化。状态证据可以是数据库记录、消息状态、Nacos 规则、Grafana Trace 或日志查询，不要求在此项目中建设自动化测试体系。
